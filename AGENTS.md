# AGENTS - MinerU 文稿 RAG 转换详细规则版

## 0. 文件定位

本文件是当前 `MinerU/` 仓库的**详细操作规范**，用于指导后续基于这个仓库执行 PDF / 扫描书稿 / 长文档的 RAG 格式转换。

它回答的不是“MinerU 源码怎么开发”，而是：

- 如何在这个仓库里稳定跑文稿解析；
- 如何把解析结果整理成真正适合 RAG 的产物；
- 如何处理跨页断句、图表、目录噪声、失败重跑和最终交付。

如果某个具体文稿目录自身也有 `AGENTS.md`，则：

- 更靠近文稿目录的规则优先；
- 本文件作为 **MinerU 工具仓库层规则** 继续生效。

---

## 1. 适用边界

### 1.1 默认适用任务

以下任务默认适用本文件：

- PDF 转 Markdown
- 扫描版 PDF OCR 后转 Markdown
- 长文档分段解析与合并
- RAG chunks 生成
- 图表清单与图表图片交付
- 文本质量巡检
- 最终 RAG 包打包与备份

### 1.2 默认不适用任务

以下任务不应默认按本文件扩展：

- MinerU 核心源码功能开发
- MinerU 模型算法优化
- 仓库级 CI / 测试体系重构
- 与文稿 RAG 转换无关的通用脚本开发

如用户明确要求改 MinerU 源码，再切换到“开发任务”思路，不要把文稿转换规则硬套进去。

---

## 2. 仓库角色与工作原则

### 2.1 仓库角色

`MinerU/` 在这类任务里默认是：

- **解析工具仓库**
- **中间产物工作区**
- **质量验证工作台**

不是：

- 文稿资产主仓库
- 最终交付物唯一归档位置

### 2.2 核心原则

1. **先保真，再清洗**
2. **先续句，再分块**
3. **先保留图表证据，再判断是否适合 RAG**
4. **先分段稳定跑通，再追求整洁输出**
5. **不把“有 Markdown”当“可直接入库”**

---

## 3. 接到任务后的标准判断

开始动手前，必须先完成以下判断，并在回复或内部执行中明确：

### 3.1 文档形态判断

- 文字版 PDF
- 扫描版 / 图片版 PDF
- 混合型 PDF

### 3.2 规模判断

- 短文档
- 中等文档
- 长文档

默认经验规则：

- 数百页书稿、教材、报告集：按长文档处理

### 3.3 内容结构判断

检查是否存在：

- 目录页
- 封面 / 扉页
- 版权页
- 图表
- 象限图
- 流程图
- 表格
- 插图

### 3.4 最终用途判断

至少区分：

- 纯文本 RAG
- 图表敏感问答
- 需要高精度引用 / 摘要

---

## 4. 环境准备规则

### 4.1 Python 环境

- 优先使用仓库内 `.venv`
- 系统 Python 版本不满足要求时，必须新建兼容虚拟环境
- 不要强行用不兼容版本运行

### 4.2 依赖安装

- 能复用就复用，不重复装
- 能复用已有模型就复用，不重复下
- 优先走最小可用依赖，不默认把所有可选后端全装上

### 4.3 模型配置

- 优先显式指定模型目录
- 优先使用本地模型配置文件
- 不要把模型路径写死到业务目录

### 4.4 运行方式

- 优先 `mineru` CLI
- 若 CLI 会自动拉起临时本地 API，要意识到它依赖回环端口绑定
- 端口绑定被权限限制时，不要误判成 MinerU 解析失败

---

## 5. 解析策略选择规则

### 5.1 文字版 PDF

执行顺序：

1. 验证能否抽到可用文本
2. 检查阅读顺序
3. 检查页眉页脚污染
4. 再决定是否需要 OCR 或额外清洗

### 5.2 扫描版 / 图片版 PDF

默认策略：

- `pipeline + ocr`

附加规则：

- 不假设能直接抽文本
- 不要先写一堆清洗脚本再发现原文根本没有文本层

### 5.3 纯文本 RAG 默认开关

默认：

- 公式解析：关
- 表格解析：关

原因：

- 先保证正文稳定
- 降低不必要模型依赖
- 降低解析时延与失败面

### 5.4 图表敏感问答

如果目标包含图表问答：

- 不允许只交纯文本 chunks
- 必须额外交付图表 manifest 与图表图片目录

---

## 6. 长文档处理规则

### 6.1 默认分段，不默认整本

长文档默认按固定页段分批跑。

推荐起始粒度：

- `64` 页一段

### 6.2 为什么

整本一次跑的典型问题：

- 失败后定位困难
- 重跑成本极高
- 临时 API / 处理窗口更容易在长任务中出错
- 一旦失败，容易前功尽弃

### 6.3 分段执行要求

- 每段独立输出
- 每段必须可单独验证
- 某段失败时只重跑失败段
- 不覆盖已成功段

### 6.4 合并前提

只有在所有分段成功后，才允许合并为整本结果。

---

## 7. 跨页 / 跨块断句规则

这是本文件最关键的硬规则。

### 7.1 禁止事项

禁止以下做法：

- 删除跨页残句
- 删除跨块续句
- 把跨页断句误判为噪声
- 生成“从半句开始”的最终 chunk

### 7.2 必做动作

在最终 chunking 前，必须做：

- 跨页续句检查
- 跨块续段检查
- 半句开头检测

### 7.3 优先判定为续接的情况

如果满足以下任一条件，优先考虑拼接：

- 上一段结尾不是完整句
- 下一段不是标题
- 下一段开头像续写
- 当前块开头是残短语、续接语气、句中从句
- 两段合起来明显更通顺

### 7.4 验收标准

最终 chunk：

- 开头尽量是标题、完整句或完整段
- 不应以句中残片起手
- 不应因硬切导致强语义断裂

如果发现 chunk 以半句开头，必须重切或重拼。

---

## 8. 图表处理规则

### 8.1 图表不是“可选附属品”

如果正文中存在：

- “见图X-X”
- 图号 / 图注
- 依赖图表理解的论述

则图表属于核心证据，不得省略。

### 8.2 最低交付要求

至少保留：

- 图号
- 图注
- 图片文件
- 图前文
- 图后文
- 图表 manifest

### 8.3 不得虚假完整

如果图内小字、坐标轴、象限标签、流程节点、细部说明没有完整 OCR：

- 必须在质量报告中明确写出来
- 不得宣称“图表信息完整”
- 不得把“保留了图注”误说成“图内信息已完整转文本”

### 8.4 最佳实践

文本 RAG 包之外，再单独交付：

- `figures.jsonl`
- `figures/`

用于：

- 图表问答
- 多模态补充检索
- 证据追溯

---

## 9. 噪声识别与过滤规则

### 9.1 默认噪声

通常应识别并过滤：

- 封面宣传语
- 扉页重复信息
- 目录页
- ISBN / 定价 / 版权 / 发行信息
- 上架建议
- 尾页营销文案

### 9.2 不能误删的内容

以下内容不能当噪声删掉：

- 跨页续句
- 正文中的模型说明
- 图注
- 图表引用上下文
- 核心案例叙述

### 9.3 目录页处理

目录页通常：

- 不应混入主向量库
- 但应保留到被过滤产物里，便于回查

---

## 10. 输出分层要求

至少保留三层结果。

### 10.1 原始解析层

示例：

- 单段 MinerU 输出
- 原始 `.md`
- `content_list.json`
- `content_list_v2.json`
- `layout.pdf`
- `span.pdf`

### 10.2 清洗整合层

示例：

- 全文清洗版 Markdown
- 续句拼接后的整本文本

### 10.3 最终 RAG 交付层

建议至少包含：

- `book.cleaned.md`
- `rag.chunks.jsonl`
- `figures.jsonl`
- `figures/`
- `rag.filtered_out.jsonl`
- `quality_report.md`
- `README.md`

---

## 11. Chunk 生成规则

### 11.1 默认目标

chunk 目标不是“长度均匀”，而是：

- 语义完整
- 检索友好
- 便于追溯

### 11.2 推荐切分顺序

优先顺序：

1. 先按页段合并后的正文块
2. 再按标题层级切
3. 再按段落切
4. 最后才按句子切

### 11.3 不推荐做法

- 不要优先按纯字符硬切
- 不要为了做 overlap 复制半句到下一块
- 不要制造以残句开头的块

### 11.4 Chunk 元信息建议

最终 chunk 建议至少包含：

- `chunk_id`
- `page_start`
- `page_end`
- `section_hint`
- `text`
- `char_count`

---

## 12. 质量检查清单

交付前必须检查：

### 12.1 文本质量

- 是否存在半句开头 chunk
- 是否存在错序
- 是否存在乱码
- 是否存在严重断裂
- 章节标题是否基本连续
- 核心内容是否顺畅读通
- 是否存在会误导召回的低质量块

### 12.2 图表质量

- 图号 / 图注是否保留
- 图片是否落盘
- `见图X-X` 是否还能追溯
- 图表关键信息是否仍依赖图片本体

### 12.3 RAG 适配性

- 是否适合直接做向量化
- 是否需要过滤目录 / 版权 / 营销块
- 是否需要多模态图表补充
- 最终推荐导入哪份文件

---

## 13. 质量报告必须回答的问题

每次交付都必须明确回答：

1. 是否**可直接用于 RAG**
2. 图表信息是否完整保留；若没有，缺口在哪里
3. 核心内容语义是否通畅
4. 是否存在会影响检索、重排、问答效果的低质量文本
5. 推荐用户实际导入哪份文件

质量结论必须显式分级：

- `可直接用于 RAG`
- `可用于 RAG 但有风险`
- `不建议直接用于 RAG`

---

## 14. 结果存放规则

### 14.1 本仓库内 output 的定位

本仓库 `output/` 默认是：

- 工具执行产物
- 中间结果
- 调试与复查材料

### 14.2 最终结果必须回传

最终可用结果：

- 不要只留在 `MinerU/output/`
- 必须回传到文稿项目目录

### 14.3 需要防误删时

额外备份到：

- `~/codex/<project>/RAG/`

备份目录建议带日期：

- `~/codex/OPC/RAG/<doc-name>-rag-package-YYYY-MM-DD/`

---

## 15. 失败处理手册

### 15.1 整本失败

如果整本任务失败：

- 不要直接重复整本命令
- 优先改成分段策略

### 15.2 单段失败

如果某段失败：

- 先保留日志
- 只重跑该段
- 不覆盖已成功段

### 15.3 质量差

若结果质量差，先判断问题来源：

- OCR 问题
- 分段问题
- 续句问题
- 噪声过滤问题
- 图表处理问题

然后针对性修正，不要把“整套重做”作为默认动作。

---

## 16. 禁止事项

- 不要把 MinerU 仓库改造成某一份文稿的专用项目
- 不要未说明风险就宣称“图表完整”
- 不要未说明风险就宣称“已经适合 RAG”
- 不要只交付一个最终 Markdown
- 不要删除跨页残句来“美化结果”
- 不要把目录页混入主向量库
- 不要把图表相关图片和 manifest 省略后还说“已支持图表问答”

---

## 17. 最终执行原则

**先稳定解析，后续句拼接，再分块入库；先保留图表证据，再判断是否真的适合 RAG。**
